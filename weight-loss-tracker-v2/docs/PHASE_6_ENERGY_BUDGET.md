# Phase 6：动态热量预算与减重计划

状态：实施中。第 1 至 5 阶段已完成，下一步扩展 AstrBot tools 和统一确认流程。

## 1. 目标

在 Phase 5 的多用户记录能力上增加自动热量计划，让用户可以通过 AstrBot 完成以下闭环：

1. 提供年龄、身高、体重和日常活动情况
2. 获取预计静息消耗和日常维持热量
3. 用户提出减重后预览静息消耗、维持热量、每日缺口和每日预算
4. 用户逐项看到数值并确认后才保存和启用计划
5. 每次进食先预览食物营养和写入后的每日预算，确认后才保存
6. 每次运动先预览运动消耗和写入后的每日预算，确认后才保存
7. 随最新体重重新计算计划，但不静默修改已生效计划

本阶段仍保持 Spring Boot 为唯一计算和数据来源。LLM 负责理解自然语言、补问缺失信息和解释结果，不负责保存或自行累计热量。

## 2. 已确认边界

本阶段不增加以下内容：

- 登录注册、公网服务和 Web 多租户鉴权
- BMI、最低摄入、目标体重或热量缺口的安全拦截
- 未成年人、疾病、怀孕等医学风险判断
- Coser 专属体脂、围度、拍摄日期或备赛字段
- 图片食物识别、营养数据库和健康设备同步
- 自动提醒、主动日报和周报推送
- 待确认操作后端持久化
- 强制迁移 MySQL 或引入新的数据库运维体系

保留普通数据合法性校验，例如年龄、身高、体重、热量和时长必须是可计算的有限非负值。BMI 仅作为已有展示指标，不参与计划生成和阻断。

### 2.1 统一确认规则

以下数值都必须先展示给用户并取得明确确认，确认前不得写入业务表：

- 静息消耗
- 基础日常消耗和维持热量
- 每日计划缺口
- 基础每日摄入预算
- 每一条食物的热量、蛋白质、脂肪和碳水
- 每一条运动的时长和消耗热量
- 本次记录写入后预计形成的当日总摄入、总消耗、剩余预算和预计缺口

用户自己明确提供数值也不能绕过确认。P6 实施后，该规则覆盖 Phase 5 中“明确提供热量可以直接写入”的插件行为。

一次确认只对应当前最新预览。预览后如果资料、活动计划或当天记录发生变化，后端必须拒绝旧预览并要求重新生成，不能使用过期数值写入。

## 3. 计算职责

### 3.1 LLM 负责

- 从聊天中提取年龄、身高、体重、日常活动和减重意图
- 区分日常非运动活动与当天专项运动
- 在缺少必要字段时向用户补问
- 将用户描述的食物转换为营养估算参数
- 将运动描述转换为运动名称、强度、时长和消耗估算参数
- 用自然语言解释后端返回的计划和每日预算

### 3.2 后端负责

- 使用固定版本的公式计算静息消耗
- 根据非运动日常活动等级计算基础日常消耗
- 计算维持热量、计划缺口、基础摄入目标和每日动态预算
- 为计划、食物和运动生成不落库的权威预览
- 汇总已确认的食物与运动记录
- 保证用户隔离、幂等、精度、舍入和日期口径一致
- 保存计划版本及其计算输入，确保结果可以追溯

LLM 不直接给数据库写入最终的维持热量和剩余额度，也不得在多轮对话中自行累计这些值。

## 4. 最小资料字段

沿用 `UserProfile` 已有字段：

- `heightCm`
- `currentWeightKg`
- `targetWeightKg`
- `dailyCalorieGoal`

新增自动计算必需字段：

- `ageYears`：当前年龄
- `formulaSex`：公式计算参数，`MALE` 或 `FEMALE`
- `nonExerciseActivityLevel`：不包含专项运动的日常活动等级
- `calorieGoalMode`：`UNSET`、`MANUAL` 或 `AUTO`

`nonExerciseActivityLevel` 第一阶段使用四档：

- `SEDENTARY`：工作和日常以坐姿为主
- `LIGHT`：日常有少量走动
- `MODERATE`：日常走动较多
- `HIGH`：日常包含较多体力活动

用户描述的跑步、力量训练、舞蹈等专项运动不进入该等级，必须按实际发生日期记录为运动，避免重复计算。

兼容规则：

- 现有非空 `dailyCalorieGoal` 视为 `MANUAL`
- 资料不足时保持 `UNSET` 或 `MANUAL`，不得伪造自动计划
- 自动计划生效后使用 `AUTO`
- 用户仍可切回手动目标

## 5. 能量计划模型

新增 `EnergyPlan`，至少保存：

- `id`
- `userId`
- `calculationMethod` 和 `calculationVersion`
- `ageYears`
- `formulaSex`
- `heightCm`
- `weightKg`
- `nonExerciseActivityLevel`
- `restingEnergyCalories`
- `baselineExpenditureCalories`
- `dailyDeficitCalories`
- `baseIntakeTargetCalories`
- `status`：`ACTIVE` 或 `SUPERSEDED`
- `effectiveFrom`
- `clientRequestId`
- `createdAt` 和 `updatedAt`

用户资料更新不会直接覆盖活动计划。重新计算先产生预览，用户确认后将旧计划标记为 `SUPERSEDED`，新计划变为 `ACTIVE`。

### 5.1 缺口来源

每日缺口按以下优先级生成：

1. 用户明确提供每日缺口
2. 用户提供目标体重和目标周期时，由后端的可版本化算法换算初始缺口
3. 信息不足时使用后端配置的默认缺口比例生成建议值

所有路径都只生成预览。预览必须同时列出静息消耗、基础日常消耗、每日缺口和基础摄入预算，用户确认整份数值后才保存并启用计划。P6 不对缺口大小设置医学拦截，但必须展示计算输入、结果和估算标识。

## 6. 统一计算口径

后端内部统一使用高精度数值计算，API 展示时再舍入为整数 kcal。

精确公式、活动系数、缺口模式、舍入和确认指纹规则见 [ENERGY_CALCULATION.md](ENERGY_CALCULATION.md)。

```text
静息消耗 = restingEnergyFormula(age, formulaSex, height, latestWeight)

基础日常消耗 = 静息消耗 × nonExerciseActivityFactor

当日运动消耗 = 当天已确认运动记录的 caloriesBurned 总和

预计当日总消耗 = 基础日常消耗 + 当日运动消耗

基础摄入目标 = 基础日常消耗 - 每日计划缺口

当日摄入预算 = 基础摄入目标 + 当日运动消耗

今日剩余可摄入 = 当日摄入预算 - 今日已摄入

当前预计缺口 = 预计当日总消耗 - 今日已摄入
```

剩余可摄入允许为负数，不能截断成零。这样用户能看到当天超过计划的具体数值。

静息消耗、每日缺口和基础摄入预算在计划确认时作为 `EnergyPlan` 的计算快照保存。当日动态预算由已确认计划和已确认记录实时推导，不单独保存一份可能过期的汇总值。

食物或运动预览必须展示“写入后的预计每日预算”。确认时后端重新计算；如果结果与预览不同，则返回 `409 Conflict` 并要求重新确认，不能先写记录再告知预算变化。

### 6.1 运动不重复计算

- 日常活动等级只包含工作、通勤和普通走动
- 跑步、健身、舞蹈等运动只通过当天运动记录加入
- 用户只描述每周运动习惯但未记录当天运动时，不将其加入当天预算
- 用户明确提供设备消耗值时沿用该值，但仍必须预览确认
- LLM 估算和用户明确提供的运动消耗使用同一套预览确认规则
- 未确认运动不影响当日总消耗和剩余预算

## 7. API 设计

### 7.1 用户资料

扩展现有接口：

```http
GET /api/users/{userId}/profile
PUT /api/users/{userId}/profile
```

请求和响应增加 `ageYears`、`formulaSex`、`nonExerciseActivityLevel` 和 `calorieGoalMode`。

### 7.2 热量计划

```http
POST /api/users/{userId}/energy-plans/preview
POST /api/users/{userId}/energy-plans
GET  /api/users/{userId}/energy-plans/active
```

- `preview` 只计算，不写入计划
- 预览返回 `previewFingerprint`，覆盖计划输入、计算结果和当前资料版本
- 创建计划必须带 `clientRequestId` 和最近一次 `previewFingerprint`
- 创建时后端重新计算并核对指纹，不一致时返回 `409 Conflict`
- 相同用户和相同 `clientRequestId` 重试时返回原计划
- 创建新计划时旧计划自动失效

### 7.3 食物和运动预览

```http
POST /api/users/{userId}/food-records/preview
POST /api/users/{userId}/exercise-records/preview
```

预览接口不写数据库，返回：

- 规范化后的待写记录和全部数值
- 数值来源和估算说明
- 写入后的预计每日预算
- `previewFingerprint`

正式创建食物或运动记录必须携带最近一次预览指纹。后端在同一事务中重新验证记录、重算预算、核对指纹并写入；任何一步不一致都不写入。

### 7.4 每日预算

```http
GET /api/users/{userId}/energy-budgets/daily?date=2026-07-12
```

响应至少包含：

- `restingEnergyCalories`
- `baselineExpenditureCalories`
- `exerciseCaloriesBurned`
- `estimatedTotalExpenditureCalories`
- `baseIntakeTargetCalories`
- `todayIntakeBudgetCalories`
- `caloriesConsumed`
- `remainingIntakeCalories`
- `projectedDeficitCalories`
- `planStatus`
- `calculationMethod`

现有每日汇总响应增加同一份 `energyBudget`，避免前端和插件各自重复计算。

## 8. AstrBot 插件

### 8.1 新增工具

- `weight_energy_plan_preview`：补齐资料后生成计划预览
- `weight_today_budget`：查询今日消耗、摄入、剩余额度和预计缺口

现有 `weight_profile_update` 扩展新资料字段。`weight_confirm` 统一确认计划、食物或运动的最近一次预览，不再增加单独的写入确认工具。

### 8.2 对话规则

用户说“我想减肥”时：

1. 查询现有资料
2. 只补问自动计算所缺字段
3. 调用计划预览
4. 返回静息消耗、维持热量、建议缺口、基础摄入预算和计算依据
5. 用户明确确认全部数值后调用 `weight_confirm` 写入计划

用户描述食物或运动时：

1. 调用后端预览接口，不写数据库
2. 展示本次食物营养或运动消耗的全部数值
3. 展示确认写入后的预计累计摄入、运动消耗、剩余可摄入和预计缺口
4. 用户明确确认后调用 `weight_confirm`
5. 后端核对预览指纹并原子写入
6. 返回实际写入值和最新每日预算

无论数值来自 LLM、用户本人还是设备，食物和运动都必须先确认。取消或预览过期时不写入任何记录，预算也不发生变化。

- 目标资料页增加年龄、计算用性别、日常活动等级和目标模式
- 仪表盘增加“预计总消耗”“今日预算”“剩余可摄入”“预计缺口”
- 目标资料页增加当前自动计划摘要和重新计算入口
- 手动目标和自动计划明确区分
- 未建计划时展示缺失字段，不显示伪造的零值
- 食物和运动 CRUD 后刷新每日预算
- Web 新增计划、食物或运动时，在最终提交前显示数值与预算变化确认框

本阶段不增加新的独立管理后台或复杂计划编辑器。

## 10. 数据迁移与兼容

- 现有用户和记录必须无损保留
- 新资料字段允许为空
- 现有手动 `dailyCalorieGoal` 行为保持不变
- 没有自动计划的用户继续使用当前每日汇总
- H2 和可选 MySQL profile 使用同一实体结构
- 自动计划表按 `user_id`、`status` 和 `effective_from` 建索引
- 不提交本地 H2 文件或迁移产生的运行数据

- 自动计划缺字段时返回明确错误
- 相同输入得到稳定的公式结果
- 四档日常活动等级计算正确
- 运动只在实际记录日期加入一次
- 未确认记录不影响预算
- 食物、运动新增和删除后预算同步变化
- 剩余摄入可为负数
- 手动和自动目标模式正确切换
- 计划预览不写数据库
- 食物和运动预览不写数据库
- 用户明确提供热量时仍不能绕过确认
- 预览后当天数据变化会使旧指纹失效并返回 `409`
- 确认失败时记录和预算均不发生变化
- 计划创建幂等，旧计划正确失效
- 两个 QQ 的资料、计划和预算完全隔离
- 最新体重变化后只提示重新计算，不静默替换计划

### 11.2 插件

- 缺少年龄或活动资料时正确补问
- 计划预览不会直接启用
- 确认后启用计划
- 明确提供的食物和运动数值同样进入待确认状态
- ToolLoop 重试不重复创建计划
- 食物确认后返回新的剩余摄入
- 运动确认后只增加一次消耗和预算
- 不同 QQ 的计划预览和确认互不干扰
- 后端不可用时不声称计划或记录成功

### 11.3 前端

- 每日预算指标展示
- 负数剩余额度展示

## 13. 完成标准

- 用户只提供最小必要资料即可生成可追溯的维持热量和减重计划预览
- 静息消耗、每日缺口和基础预算未经确认不会保存或生效
- 每一条食物和运动，无论数值来源，都未经确认不会写入
- 每次预览同时展示确认后的预计每日预算变化
- 饮食确认后可以准确返回今日剩余可摄入热量
- 运动确认后只计算一次，并同步更新总消耗和可摄入额度
- 过期预览不能写入，确认过程不会产生部分成功
- 手动热量目标继续可用
- 体重变化不会静默修改计划
- 两个 QQ 用户的计划与预算完全隔离
- 所有自动计算由后端完成，Web 和插件不包含第二套公式
- 数据、测试、Docker 和 AstrBot 运行验证全部通过
